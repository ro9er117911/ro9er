### Design direction

```text
Enterprise creative portfolio
```

第一眼要像作品集，有視覺衝擊；第二眼才讓人知道 Roger 處理 AI / 金融 / 城市 / 系統風險；第三眼提供作品、影片、知識圖譜互動。

### Knowledge Graph

Knowledge Graph 是輔助探索工具，放在 Featured Work 後面，不當首頁主入口。

對外文案：

```text
Click a node. See how the work connects.
點一個節點，看作品、經歷和能力怎麼連在一起。
```

不要顯示：

```text
NetworkX-inspired thinking, rendered as static SVG/CSS for GitHub Pages. The point is not the library; the point is the operating model.
```

NetworkX 是 Python package，可作為資料建模來源或內部實作概念，但不是使用者需要看的訊息。前端呈現應該像輕量圖資料庫：節點、邊、點擊後更新說明面板。

### Theory proof from AI governance paper

AI governance 論文提供網站的理論框架，不搶主視覺。核心語言：

```text
AI changes the responsibility chain: who frames the task, who reviews it, who owns residual risk, and what evidence remains after a decision is made.
```

公開呈現時避免過度聲稱實證結果。可說這是 governance design / deterministic probe / responsibility-chain theory，不說它已證明銀行真實成效。

0. 設計判斷

目前版本方向是正確的：深色、大片 Hero、AI systems builder、Selected Work、Method、Contact。問題是互動性仍偏「靜態展示」，觀者只能往下滑，不能快速理解你的跨域能力如何組成一個系統。

新版目標不是加花俏動畫，而是增加「可操作的理解路徑」。你的核心不是單一職稱，而是跨 AI、FinTech、治理、防災、城市資料、Agent workflow 的高風險決策系統建構者。這與你文件中的定位一致：AGENT.md 把你的使命定義為用 AI、資料科學、圖形演算法去預測風險、建立防禦架構、最大化保護效益。 SOUL.md 則把你的核心價值收束在風險抗性、跨界重組、技術的溫度、守正出奇。

DESIGN.md 原本已經確立：這不是 generic resume page，而是要呈現你作為 pragmatic AI / data / product systems builder。第一屏要說清楚你做什麼、哪些 proof points 值得檢查、如何快速聯絡或評估你。 所以 v2 應該從「作品列表」升級成「互動式證據控制台」。

1. Product Goal

建立一個 GitHub Pages-ready、靜態可部署、互動感強、直觀可掃描的個人作品集。

核心功能：

讓訪客在 10 秒內知道你是誰。
讓訪客在 30 秒內理解你的 4 個能力域。
讓訪客在 90 秒內檢查 1 個代表作品的證據。
讓教授、業界主管、技術同儕、合作者可以各自用不同入口理解你。

不做：

不做 CMS。
不做後端。
不做登入。
不做資料庫。
不做炫技型 3D 網站。
不做需要伺服器才能跑的 AI chatbot。

這符合原本 DESIGN.md 對 GitHub Pages 的限制：第一版建議用純靜態 HTML、CSS、少量 JavaScript，避免 backend、CMS、authentication 和不必要 build step。

10.3 UI 表現

左側像聊天 prompt，右側是事件 log：

User asks
→ Parser scopes intent
→ Data tool selects dataset
→ UI event moves map
→ Evidence card appears

這比放一張 dashboard 截圖更有說服力，因為它讓觀者理解你設計的是「資料工具層 → 事件層」的互動系統。
